## 关系抽取原则

### 1. 实体分类收紧，关系表达放松

实体类型应尽量稳定，否则后面统计、筛选、建图都会失去一致性。  
但关系表达应尽量开放，因为文化语义常常依赖上下文，不适合过早标准化。

例如：

- `康有为` 与 `丹灶` 的关系，可以是“出生于”
- `黄飞鸿` 与 `武术文化` 的关系，可以是“代表人物”
- `松塘村` 与 `古建筑群` 的关系，可以是“保存有”
- `《南海县志》` 与 `西樵山` 的关系，可以是“记载”

这些都不必在抽取时强行改写成同一种通用关系。

### 2. 先保留原始关系语义，再做后置规整

开放式抽取的重点，不是马上做统计，而是先尽量保留文化信息。  
因此每条关系至少保留三层信息：

- `source`：源实体
- `relation_text`：原始关系表达
- `target`：目标实体

若条件允许，最好再附带：

- `evidence`：原文证据
- `source_text`：来源文本
- `confidence`：置信度
- `source_file`：来源文献

这样即使后面要重分类，也不会丢失最初的文化语义。

### 3. 关系短语应尽量使用文本中真实语义

关系抽取时，不要求大模型只能从固定列表里选，而是要求它：

- 用简洁、明确、可读的短语表达关系
- 优先使用文本本身能支持的语义
- 避免使用过度抽象、过度泛化的表达

例如优先：

- `出生于`
- `流传于`
- `创建`
- `捐建`
- `记载于`
- `题咏`
- `祭祀于`
- `盛行于`
- `修建于`
- `代表`

而不宜大量出现：

- `相关`
- `有关`
- `联系`
- `涉及`

因为这些关系过于模糊，不利于后续图谱分析。

### 4. 关系开放，但不能失控

开放式关系抽取不等于完全无边界。  
仍然需要给模型一些约束，以避免关系泛滥：

1. 只抽取文本中有明确依据的关系。
2. 关系必须是动词性、判断性或短语性表达。
3. 不抽取空泛关系，如“相关”“体现”“反映”。
4. 尽量避免一长串解释句充当关系名。
5. 同一对实体可存在多条关系，但应分别表达。











































## 一、调整方向

这套分类体系的重点应当放在“文化层”本身，而不是先按景点、旅游业态或行政管理口径来分。  
因此分类逻辑应满足以下原则：

1. 以文化本体为核心，优先突出非遗与文化遗产。
2. 非遗体系单列为第一核心，不能被并入笼统的“民俗”或“文化活动”。
3. 文物、古建筑、遗址等物质文化遗产单列为第二核心。
4. 人物、宗族、空间、文献、时序属于支撑维度，服务于文化本体分析。
5. 分类与“属性”分开处理，避免层级混乱。

## 二、推荐总结构

建议采用“三层结构 + 六大一级类”：

### 第一层：文化本体层

A. 非遗文化体系  
B. 物质文化遗产体系

### 第二层：文化传承与承载层

C. 传承主体体系  
D. 文化空间体系

### 第三层：文化认知与组织层

E. 文献记忆体系  
F. 历史时序体系

其中：

- `A 非遗文化体系` 与 `B 物质文化遗产体系` 是核心主类。
- `C-D-E-F` 是支撑类，用于解释“谁传承、在哪里承载、如何被记载、处于什么历史阶段”。

## 三、一级与二级分类方案

### A. 非遗文化体系

这是第一重点，应放在整个体系的最前面。

适用于戏曲、曲艺、音乐、舞蹈、表演性文化项目。

如：

- 醒狮
- 粤剧
- 龙舟说唱
- 十番音乐

适用于手工技艺、生产技艺、工艺制作等。

如：

- 九江双蒸酿制技艺
- 灰塑
- 藤编
- 木雕
- 砖雕
- 陶塑

#### A3 民俗节庆类非遗

适用于具有集体性、节令性、地方庆典性的文化活动。

如：

- 生菜会
- 龙舟竞渡
- 庙会
- 花灯会

#### A4 信俗礼仪类非遗

适用于祭祀、信仰、诞会、地方礼俗等。

如：

- 民间信仰活动
- 祭祀仪式
- 龙母诞
- 地方礼俗

#### A5 传统体育游艺类非遗

适用于武术、竞技、游艺、身体实践类文化项目。

#### A6 饮食酿造类非遗及文化物产

不是所有食物都能归入这里，只有同时具备“传统工艺、地方记忆、文化符号”三者之一的对象，才纳入文化体系。

- 九江双蒸酒

- 煎堆技艺
- 鱼花习俗及相关物产

## 四、需要特别避免的层级混乱

为了保证体系合理，以下内容建议不要直接作为并列分类：

### 1. 文保级别不要作为类别

像“省保单位、市县保单位、历史建筑点位”这类内容，更适合作为属性字段，而不是分类层级。  
也就是说：

- `古祠堂` 是类别
- `省级文保单位` 是属性

否则会把“对象是什么”和“它被如何认定”混在一起。

### 2. 行政级别不要作为文化类别

如“区级、镇级、村级”，应作为空间属性，而不是文化分类。

### 3. 旅游业态不要混入文化本体层

如：

- 景区
- 公园
- 商业街
- 娱乐设施

这些应属于景点谱系或旅游产品层，不应直接放进文化分类体系主干。

### 4. 现代设施不要与文化遗产并列

现代纪念馆、现代展馆、现代公园可以作为“文化展示载体”或“景点产品”，  
但不能直接与非遗、古建筑、古遗址放在同一文化本体层级中。

## 五、建议采用的最终版本

如果要形成一套最稳妥、最适合论文和代码同时使用的版本，建议最终定为：

### 一级分类

- A 非遗文化体系
- B 物质文化遗产体系
- C 传承主体体系
- D 文化空间体系
- E 文献记忆体系
- F 历史时序体系

### 核心层级关系

- `A-B` 是文化本体层
- `C-D` 是传承承载层
- `E-F` 是认知支撑层

### 研究重点顺序

1. 非遗文化体系
2. 物质文化遗产体系
3. 传承主体体系
4. 文化空间体系
5. 文献记忆体系
6. 历史时序体系

## 六、结论

这套体系相比原来的简单分类，更适合南海区文旅融合研究，原因在于：

1. 把非遗放到最核心位置，符合你强调的重点。
2. 明确区分“文化本体”与“支撑维度”，层级更清晰。
3. 把“古村落”这类兼具空间性与遗产性的对象单独考虑，避免误归类。
4. 把“文保级别、行政级别、旅游业态”从分类中剥离，避免体系混乱。
5. 后续无论是做论文、NER、知识图谱还是双谱系耦合，这套体系都更容易落地。

## 七、面向知识图谱的关系体系调整建议

如果研究目标是抽取大量文化实体，并进一步构建 Neo4j 知识图谱，那么关系体系不宜一开始就限定为少数固定类型。  
原因在于：文化实体之间的语义联系本身就是多样、细密、开放的，过早压缩关系，会导致文化语义丢失。

因此建议采用：

**分类体系稳定化，关系体系开放化，后期再做归并。**

也就是说：

- 实体类型可以先按本文件的六大体系稳定下来。
- 关系不必在抽取阶段强行限制为固定十几种。
- 先让模型按文本真实语义输出关系短语。
- 再在整理阶段把关系归并为少数“上位关系类”。

这样更适合文化研究型知识图谱，而不是纯工程化、强约束型知识图谱。

## 八、关系抽取的推荐原则

### 1. 实体分类要“收”，关系表达要“放”

实体类型应尽量稳定，否则后面统计、筛选、建图都会失去一致性。  
但关系表达应尽量开放，因为文化语义常常依赖上下文，不适合过早标准化。

例如：

- `康有为` 与 `丹灶` 的关系，可以是“出生于”
- `黄飞鸿` 与 `武术文化` 的关系，可以是“代表人物”
- `松塘村` 与 `古建筑群` 的关系，可以是“保存有”
- `《南海县志》` 与 `西樵山` 的关系，可以是“记载”

这些都不必在抽取时强行改写成同一种通用关系。

### 2. 先保留原始关系语义，再做后置规整

开放式抽取的重点，不是马上做统计，而是先尽量保留文化信息。  
因此建议每条关系至少保留三层信息：

- `source`：源实体
- `relation_text`：原始关系表达
- `target`：目标实体

若条件允许，最好再附带：

- `evidence`：原文证据
- `source_text`：来源文本
- `confidence`：置信度
- `source_file`：来源文献

这样即使后面要重分类，也不会丢失最初的文化语义。

### 3. 关系短语应尽量使用文本中真实语义

关系抽取时，不要求大模型只能从固定列表里选，而是要求它：

- 用简洁、明确、可读的短语表达关系
- 优先使用文本本身能支持的语义
- 避免使用过度抽象、过度泛化的表达

例如优先：

- `出生于`
- `流传于`
- `创建`
- `捐建`
- `记载于`
- `题咏`
- `祭祀于`
- `盛行于`
- `修建于`
- `代表`

而不宜大量出现：

- `相关`
- `有关`
- `联系`
- `涉及`

因为这些关系过于模糊，不利于后续图谱分析。

### 4. 关系开放，但不能失控

开放式关系抽取不等于完全无边界。  
仍然需要给模型一些约束，以避免关系泛滥：

1. 只抽取文本中有明确依据的关系。
2. 关系必须是动词性、判断性或短语性表达。
3. 不抽取空泛关系，如“相关”“体现”“反映”。
4. 尽量避免一长串解释句充当关系名。
5. 同一对实体可存在多条关系，但应分别表达。

## 九、关系体系的两阶段方案

### 第一阶段：开放式关系抽取

这一阶段的目标是“尽量保留文化语义”，不急着做统一标准。

推荐输出格式：

| 字段 | 含义 | 说明 |
|------|------|------|
| source | 源实体 | 必须是已识别实体 |
| relation_text | 原始关系短语 | 由模型根据文本自行判断 |
| target | 目标实体 | 必须是已识别实体 |
| evidence | 原文证据 | 最好保留原句或短证据 |
| confidence | 置信度 | 便于后续筛选 |
| source_file | 来源文件 | 便于溯源 |

这一阶段允许出现大量细粒度关系，如：

- 出生于
- 流传于
- 发源于
- 保存于
- 建于
- 重修于
- 捐建
- 祭祀于
- 记载于
- 收录于
- 题咏于
- 倡导
- 代表
- 师承
- 依附于
- 盛产
- 属于
- 演变为

这些都可以先保留，不急于压缩。

### 第二阶段：关系归并与上位类整理

这一阶段的目标是“服务统计分析与 Neo4j 建模”。  
也就是说，前面开放抽取出来的细粒度关系，后面可以再整理为若干关系大类。

建议采用“原始关系 + 归并关系”双字段结构：

- `relation_text`：保留原始语义
- `relation_group`：用于后期分析的上位类

这样你既可以看到具体文化语义，又能做结构化分析。

## 十、建议采用的关系上位类

虽然不建议在抽取阶段硬限制关系种类，但为了后续建图与分析，仍建议预设若干“关系上位类”。

### 关系上位类总表

| 关系上位类 | 含义 | 常见原始关系示例 | 适用对象 |
|------|------|------|------|
| 空间关联类 | 表示文化对象与地方空间之间的联系 | 位于、流传于、发源于、活动于、分布于、盛行于 | 非遗、人物、建筑、遗址、事件、物产 |
| 传承延续类 | 表示文化的传承、继承、守护与延续 | 传承、师承、守护、延续、代表、承袭 | 非遗、人物、宗族、社群 |
| 营建创造类 | 表示建造、修缮、捐建、设立等关系 | 创建、修建、重修、捐建、设立、营建 | 人物、宗族、建筑、遗址 |
| 文献记载类 | 表示文献记录、题咏、收录、记述等关系 | 记载于、见于、收录于、题咏于、载于 | 文献与各类文化实体 |
| 时序演变类 | 表示时间归属、发生、兴盛、演化等关系 | 属于时期、发生于、形成于、兴盛于、演变为 | 事件、非遗、建筑、人物 |
| 从属分类类 | 表示类别归属与体系位置 | 属于、归入、细分为、同属 | 各类实体与分类节点 |
| 文化表征类 | 表示人物、建筑、地方对某文化的代表性 | 代表、象征、体现、承载、展示 | 人物、建筑、空间、非遗 |
| 社群组织类 | 表示宗族、社群、地方组织与文化对象的关系 | 属于某宗族、由某社群主持、由某班社传承 | 人物、宗族、非遗、建筑 |
| 产品转化类 | 表示文化资源与旅游展示、体验产品之间的关系 | 转化为、展示于、开发为、依托于 | 文化实体与景点、展馆、基地 |

这里要注意：

- `relation_group` 是后期归并分析用的。
- `relation_text` 仍然保留原始关系表达。
- 不建议只保留 `relation_group`，否则会损失文化细节。

## 十一、面向 Neo4j 的推荐建模方式

如果你的目标是把抽取结果尽量自然地落入 Neo4j，推荐采用“节点稳定、边双层表达”的策略。

### 1. 节点层

节点可以继续沿用本文件的六大实体体系：

- 非遗文化体系
- 物质文化遗产体系
- 传承主体体系
- 文化空间体系
- 文献记忆体系
- 历史时序体系

### 2. 边层

每条边建议同时保留：

- `relation_text`
- `relation_group`
- `evidence`
- `source_file`
- `confidence`

即使 Neo4j 的关系类型最终采用相对收敛的上位类，也建议把原始关系短语作为关系属性保留下来。

例如可以理解为：

- 边类型：`空间关联类`
- 边属性中的 `relation_text`：`流传于`

或者：

- 边类型：`文献记载类`
- 边属性中的 `relation_text`：`记载于`

这样能兼顾图结构整洁与文化语义保留。

## 十二、适合本项目的最终建议

结合你目前“抽取大量文化实体并制成 Neo4j 知识图谱”的目标，最合适的方案不是：

- 关系完全固定化

也不是：

- 关系完全无约束自由化

而是下面这种折中且更稳妥的方案：

### 推荐工作流

1. 先按本文件的六大体系稳定实体分类。
2. 抽取阶段让大模型开放输出 `relation_text`。
3. 过滤掉过于模糊的关系表达。
4. 再把原始关系归并为 `relation_group`。
5. 最后将“节点类型 + 原始关系 + 关系上位类 + 证据”一起写入 Neo4j。

### 这样做的好处

1. 不会在前期丢失文化关系的细节。
2. 可以保留南海地方文化语义的复杂性。
3. 后期仍然可以做统计、可视化、聚类和图谱分析。
4. 更适合你的研究目标，而不只是适合程序实现。

## 十三、最终结论

对于本项目来说，最好的分类体系完善方向应当是：

- **实体分类采用稳定体系**
- **关系抽取采用开放策略**
- **关系分析采用后置归并**

这样才能同时满足：

1. 大规模文化实体抽取的可操作性
2. 文化关系语义表达的丰富性
3. Neo4j 图谱建模的结构性
4. 后续论文写作与统计分析的可解释性

因此，这份分类体系不仅应当是一份“实体分类表”，还应当是一份能够指导：

- 实体抽取
- 关系抽取
- 关系归并
- 图谱建模
- 文化分析

的基础框架。
